Inversion Of Control

Wat een hippe term, "Inversion of Control". Maar wat betekent het?

Inversion of Control (IoC) betekent dat objecten niet verantwoordelijk zijn voor de creatie van andere objecten waar zij een afhankelijkheid van hebben. In plaats daarvan krijgen zij deze objecten van een externe bron (bijv. een XAML config file).

Neem als voorbeeld een View Model in een WPF applicatie. Dit View Model moet vaak zijn data vaak uit een persistent store halen zoals een database. De verantwoordelijkheid voor het ophalen van deze data zit dan bijvoorbeeld in een Repository.

Example: Repository in Viewmodel

 
public class MainViewModel : ViewModelBase
{
    private ICarRepository _carRepository;

    public ObservableCollection<CarVM> MyCars { get; set; }

    public MainViewModel(ICarRepository carRepository)
    {
        this._carRepository = new DummyCarRepository();
        var cars = _carRepository.GetAllCars().Where(c => !c.Actief).ToList();
        this.MyCars = new ObservableCollection<CarVM>(cars.Select(c => new CarVM(c)));
    }
}

Volgens de principes van IOC mag er nu geen instantie van het interface ICarRepository gemaakt worden in de klasse MainViewModel. De verantwoordelijkheid voor deze initialisatie moet naar een andere klasse worden verplaatst.

IOC is dus het verplaatsen van verantwoordelijkheid. Een manier om dit te doen is Dependency Injection.

Disclaimer

Het principe achter IOC is zeer breed toepasbaar. De toepassingen van IOC zul je daarentegen bijna altijd tegenkomen via de implementatie van een ander software patroon. Het nut van deze patronen word pas duidelijk bij het bouwen van een grootschalige software applicatie. De voorbeelden tijdens de workshops zullen nooit zo groot zijn dat het software patroon echt tot zijn recht komt.
